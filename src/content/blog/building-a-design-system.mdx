# A single-accent teal design system

Most design systems don't fall apart for lack of tokens. They fall apart because six months in, nobody remembers where a color is supposed to come from, so someone inlines a hex, someone copies it, and the palette drifts into soup.

This site is my attempt to not let that happen: one accent color, a fixed set of tokens, and a small pile of reusable classes. No dark mode, no second accent, no rainbow gradients. The whole thing fits in two CSS files.

---

## One accent color, on purpose

It's a light theme built around a single light-teal accent, `#5EC8C0`, plus a darker hover (`#45B3AB`), a soft tint for fills (`#DBF1EF`), and an "ink" shade for text on that tint (`#1F706A`). That's the color.

The accent shows up in links, the primary button, list markers, stat numbers, and the hero grid, and nowhere else. Everything else is neutral surfaces and three levels of text:

- `--sl-text-primary` `#1A2B2A` for body and headings
- `--sl-text-secondary` `#566766` for supporting copy
- `--sl-text-muted` `#8A9897` for captions and metadata

There are three gradients, all single-hue, used for emphasis only: `--sl-gradient-accent` for buttons, `--sl-gradient-text` for accent words and stats, `--sl-gradient-soft` for tinted sections. Smear one across a whole section and it stops reading as emphasis.

## Two layers of tokens

The site's on **Tailwind v4**, so there's no `tailwind.config.js`. Everything lives in [`src/styles/global.css`](src/styles/global.css), declared in two mirrored places.

The `@theme` block turns custom properties into utilities automatically:

```css
@theme {
  --color-sl-accent: #5ec8c0;   /* -> bg-sl-accent, text-sl-accent, border-sl-accent */
  --font-display: "Sora", "Inter", system-ui, sans-serif; /* -> font-display */
  --radius-sl-lg: 20px;         /* -> rounded-sl-lg */
  --shadow-sl-md: 0 4px 16px rgba(26, 43, 42, 0.08); /* -> shadow-sl-md */
}
```

So in markup I write `bg-sl-surface`, `rounded-sl-lg`, `shadow-sl-md`, and the values are guaranteed to come from the token set.

Then `:root` mirrors those and adds what isn't exposed as a utility: gradients, the fluid type scale, spacing, the nav height. Those get used inside component CSS where a plain `var()` is nicer.

```css
:root {
  --sl-accent: #5ec8c0;
  --sl-gradient-accent: linear-gradient(135deg, #6fd3cb 0%, #3fa9a1 100%);
  --sl-text-hero: clamp(2.5rem, 6vw, 4.5rem);
  --sl-nav-height: 88px;
}
```

The split: **utilities in markup, `var(--sl-*)` in component CSS.** First rule of the whole thing is don't inline a raw hex, radius, or font. If something's missing, add a token here.

## The components

A handful of `.sl-*` classes do the repeating, so I never rebuild the same button or card twice. They're all built from tokens, which is what lets me reskin the site from one place. A few of the ones I reach for most:

**Buttons** come in two flavors, primary (the accent gradient) and secondary (outline):

```html
<a class="sl-btn sl-btn-primary" href="/work">See my work</a>
<a class="sl-btn sl-btn-secondary" href="/cv">Read the CV</a>
```

**Badges** are pill tags on the soft tint:

```html
<span class="sl-badge">Astro</span>
<span class="sl-badge">TypeScript</span>
```

**Stats** pair a gradient number with a muted label:

```html
<div class="sl-stat-number">2M+</div>
<div class="sl-stat-label">Images made</div>
```

**Accent text** drops the teal gradient onto a single word in a heading:

```html
<h1>Transform any face <span class="sl-accent-text">instantly</span></h1>
```

And **cards** are just tokens stacked up, with a hover lift:

```css
.sl-card {
  background: var(--sl-bg-surface);
  border: 1px solid var(--sl-border);
  border-radius: var(--sl-radius-lg);
  padding: var(--sl-space-8);
  box-shadow: var(--sl-shadow-sm);
}
.sl-card:hover {
  border-color: var(--sl-border-accent);
  box-shadow: var(--sl-shadow-md);
  transform: translateY(-2px);
}
```

Since this is MDX, I can drop the real `Card`, the exact one from the home page, straight into the post:

<Card title="A component, mid-article">
  Imported with a relative path and rendered inline. No copied markup, no
  one-off styles. It looks right because the system is the only thing it's
  made of.
</Card>

There are layout helpers too (`.sl-container`, `.sl-section`, `.sl-section-tint`, `.sl-eyebrow`) plus the sticky pill nav and hero band. The habit: compose these before writing new CSS, and when I genuinely need something new, add one more token-driven class in the same style.

## The hero band trick

The home page hero sits in a light-teal band that stretches *up behind* the floating nav, so the nav looks like it's hovering over the header. The whole illusion is one token doing triple duty. `--sl-nav-height` (88px) is the nav's height, the distance the band pulls itself upward, and the `scroll-padding-top` for anchor links:

```css
.sl-band {
  background: var(--sl-gradient-header);
  margin-top: calc(-1 * var(--sl-nav-height));
}
```

The home page passes `padForNav={false}` so the full-bleed hero slides under the nav; every other page keeps the default `true`. Bump the nav height and the overlap, spacing, and scroll offset all move with it.

---

## Posts get it for free

Blog posts are MDX, and the body styling comes from a single `.sl-prose` class in [`src/styles/prose.css`](src/styles/prose.css). That's why this paragraph, the `inline code`, the blockquote, and the lists all match the rest of the site with zero per-post styling.

> Blockquotes get the teal left-border treatment, so a callout reads like part of the system instead of a browser default.

The deal: never reach for inline styles. Write prose, drop in a component when prose isn't enough, and `.sl-prose` handles the rest.

---

## What it won't do

No dark mode, no second accent, no swapping the CSS framework, no UI runtime like React or Vue on the static pages. Each of those would add more places a color or component could live, and that drift is exactly what the system exists to prevent.
